### ⚠️ Открытый вопрос Фазы 2 — радиус сплэша (нужно решение ГД/пользователя)
Приёмка №10 GDD: «толпа Мечников проигрывает Камнемёту, 1 Рыцарь побеждает». **Механика корректна, но `splashRadius:1.8` из GDD слишком мал для нашей раскладки.** Замер: при столкновении мелейщики кучкуются на **~3.4** ед. друг от друга (не 1.8), поэтому сплэш ловит только основную цель → 5 Мечников **побеждают** Камнемёт (треугольник не замкнулся). Рыцарь-соло проходит (сплэш по 1 цели = обычный урон). Варианты фикса (балансный проход): (а) поднять все радиусы сплэша ~×2 (brute/mage/Камнемёт 1.8→~3.5-4, kamikaze 2.5→~5); (б) ужать боевую раскладку (`combatHoldRadius`/спавн-Y), чтобы толпа стояла плотнее; (в) принять как есть. Числа GDD не менял самовольно — жду решения.

## Журнал решений по ходу
- **Фаза 1:** `createBattleUnit` оставляет `attackRange` для melee = baseAttackReach+meleeBonus (падение в дефолт), для ranged — из класса (лучник 12). Врагам `attackRange` в данных волны = абсолютное значение (враж. лучник Лок.2 = 12).
- **Фаза 1:** для Фазы 1 в waves.json только 2 волны (1-1, 1-2). Победа = зачистка последней → при 2 волнах игра завершается быстро, это ожидаемо; остальные волны добавляются с механиками.
- **Кэш:** добавлен `cache:"no-store"` + `tools/nocache_server.py`; dev-хук `window.gameState` в main.js.
