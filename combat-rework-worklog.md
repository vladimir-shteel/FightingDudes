### Фаза 3 — сделано и проверено
- **3.1 Категории коллизий по movementType** (`battlePhysics.js`): +`CATEGORY_FLYING_ALLY/ENEMY`; `getCollisionFilter()` — ground: стены+свой наземный ряд; flying: стены+свои летающие (нет коллизии с наземными); kamikaze: только стены (проходит сквозь всех). Стены ловят все мобильные категории.
- **3.2 Фильтр canHitFlying** (`chooseTarget`): не-canHitFlying исключают летающих из кандидатов ДО keepCurrent/closest. Проверено: 3 Мечника vs 1-4 → поражение (Орла не достать, targetHint "advance"); 2 Мечника+2 Лучника vs 1-4 → победа (Лучники сбили Орла).
- **3.3 Камикадзе** (`updateKamikazes` в tickBattle, обе стороны): targetMode backline (ближайший из ряда "back", иначе ближайший); attackRange 0 (едет до контакта); при getBodyGap < `kamikazeContactGap` (0.5) — AoE `explosionDamage` по `explosionRadius` + splash-ring + самоуничтожение; в applySideAttacks камикадзе пропускается. Проверено синтетически: прошёл сквозь переднего Мечника (тот цел), взорвал тыловбого Лучника, сам погиб.
- **3.4 Рендер полёта** (`battlefield.js placeToken`): letающие рисуются на `y − flyHeight` (конфиг `flyHeight`=5), `.is-flying` — подъём, bob-анимация иконки, приглушённая тень (0.32), z-index 4. Проверено замером: top 2.9% при y=5.9, анимация flying-bob активна.
- **3.5 Волны 1-4/1-6/1-7** в waves.json (теперь 7 волн Лок.1): 1-4 «Первый полёт» (2 Дикаря + Орёл), 1-6 «Первый паззл» (4 Крысы front + Медведь back), 1-7 «Вождь племени» босс (3 Дикаря+Камнемёт front / Вождь isBoss back / Орёл flying). Спавн всех групп/боссов проверен, бои резолвятся без ошибок консоли.
- Конфиг-ручки: `battle.flyHeight` (5), `battle.kamikazeContactGap` (0.5).
- Примечание: Камнемёт в 1-7 оставлен 80/10 (минион босс-волны), не 200/12 как соло-пазл 1-5. Камикадзе-волн в Локации 1 нет — механика проверена синтетически, боевое применение в Локации 2 (колесница, волна 2-3).

### ⚠️ Открытый вопрос Фазы 2 — баланс треугольника (приёмка №10)
Приёмка №10 GDD: «толпа Мечников проигрывает Камнемёту, 1 Рыцарь побеждает». Механика сплэша корректна, но **треугольник не замыкается геометрией**.

**Ручки вынесены в конфиг (коммит 5ead41c):** `battle.splashRadiusMultiplier` (глоб. множитель радиуса, дефолт 1), `battle.spawnSpreadPadding` (стартовый разброс Y, дефолт 3.5), `window.CONFIG` — правится наживую из консоли.

**Ключевой вывод (проверено живыми боями):** дело НЕ в раскладке/радиусе. При Камнемёте 80 HP даже `mult=3.5` + юниты в кучу (`pad=13.5`) → толпа из 5 Мечников ПОБЕЖДАЕТ за ~8с. Причина: 5 Мечников = ~35 dps, Камнемёт 80 HP живёт в бою ~2.3с и физически не успевает нанести достаточно AoE-залпов (юниту 32 HP нужно ~4 залпа по 10). Недостающий рычаг — **живучесть Камнемёта** (статы в waves.json, уже data-driven).

**РЕШЕНО (одобрено пользователем, применено):** `splashRadiusMultiplier = 2` (balance.json) + Камнемёт **200 HP / 12 atk** (waves.json, волна 1-5). Проверено на реальной волне с закоммиченными дефолтами: 5 Мечников проигрывают (11.1с), 1 Рыцарь ур.3 побеждает выжив (18.1с). **Приёмка №10 ✅ замкнута.** Множитель 2 глобальный — усиливает и AoE своего Громилы/Мага (их роль). Примечание: будущий Камнемёт в волне 1-7 (Фаза 3) сделать такими же статами для консистентности.

## Журнал решений по ходу
- **Фаза 1:** `createBattleUnit` оставляет `attackRange` для melee = baseAttackReach+meleeBonus (падение в дефолт), для ranged — из класса (лучник 12). Врагам `attackRange` в данных волны = абсолютное значение (враж. лучник Лок.2 = 12).
- **Фаза 1:** для Фазы 1 в waves.json только 2 волны (1-1, 1-2). Победа = зачистка последней → при 2 волнах игра завершается быстро, это ожидаемо; остальные волны добавляются с механиками.
- **Кэш:** добавлен `cache:"no-store"` + `tools/nocache_server.py`; dev-хук `window.gameState` в main.js.
