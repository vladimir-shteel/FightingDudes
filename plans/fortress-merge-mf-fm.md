# План: Прототип Fortress Merge + Merge Factory

Рабочая ветка для интеграции поверх существующего прототипа: `codex/fortress-merge-from-master`.
Отдельная ветка для эксперимента "с нуля": `experiment/fortress-merge-mf-fm`.
Источник требований: `Промты MF+FM.txt` (Промт 1).

## 0. Цель и стратегия

Цель — не переписать игру с пустого листа и не просто приклеить новый экран к старому прототипу. Цель — **переделать существующий master-прототип под новое видение Fortress Merge + Merge Factory**, максимально переиспользуя уже работающие части.

Что уже есть в `master` и должно быть переиспользовано:
- экономика ресурсов, шахты, слоты рабочих и прогресс добычи;
- покупка/мердж человечков;
- подготовка юнитов и часть боевой математики/визуальных эффектов;
- общий статический стек GitHub Pages, JSON-конфиги, game loop и UI-рендер.

Что должно измениться принципиально:
- верхняя часть игры становится не старым battlefield/bridgehead, а **полем застройки 5×7 в стиле Fortress Merge**;
- кнопка «В бой» запускает бой **на поле застройки**, где враги атакуют здания, а здания/союзники защищают штаб;
- нижняя часть игры становится **экраном производства ресурсов**, основанным на текущих шахтах и мердже, но приведённым к требованиям промта;
- старые термины и UI (`Fighting Dudes`, `Garrison`, `Bridgehead`, бой против замка) постепенно заменяются на логику и подачу Fortress Merge + Merge Factory.

Иными словами: **нижнюю экономику адаптируем, верхний build-field и field-battle строим заново, старый бой используем только как источник подходов/утилит, а не как финальную механику**.

Не делаем в этой ветке:
- не удаляем существующий master-прототип ради "чистого" старта;
- не оставляем старый battlefield как основной бой;
- не создаём два независимых режима, которые выглядят как две разные игры;
- не ломаем работающую экономику до появления функциональной замены.

Идея конечного прототипа: два экрана, свайп вверх/вниз переключает между ними.
- **Верхний экран** — поле застройки 5×7 тайлов, покупка/размещение/апгрейд зданий, кнопка «В бой», волны врагов.
- **Нижний экран** — куча человечков-рабочих, мердж, 4 шахты (дерево/камень/металл/кристаллы), генерация ресурсов.

Ресурсы целевой модели: `wood`, `stone`, `iron`, `crystal`, `gold`. В текущем master-прототипе ресурс камня называется `ore`; при интеграции либо переименовываем его в `stone` миграционным шагом, либо временно используем `ore` как внутренний ключ камня с явным UI-label `Stone`.

Игровой цикл: копим ресурсы на нижнем экране → покупаем/размещаем/апгрейдим здания на верхнем → «В бой» → победа даёт золото + выбор апгрейда из 3 случайных → следующая волна. 7 волн всего.

## 1. Общая архитектура кода

Работаем от `master`, где уже есть живой прототип. Старые файлы (`index.html`, `js/game/*`, `data/*.json`, `styles.css`) не считаются мусором: сначала читаем и адаптируем их. Удаляем или переименовываем существующий код только когда новая реализация уже покрывает ту же роль. Инфраструктура та же: статический сайт для GitHub Pages, JSON через `fetch()`, данные в JSON, правила в системах, UI отдельно.

Рекомендуемая стратегия миграции:
1. Зафиксировать текущие зоны ответственности master-прототипа: ресурсы/шахты/рабочие, резерв/мердж, бой, UI.
2. Добавить целевое состояние Fortress Merge рядом с существующим состоянием, не ломая нижнюю экономику.
3. Переложить UI в два экрана: верхний build-field, нижний production. Старые `Mines/Reserve` становятся основой нижнего экрана.
4. Заменить старый battle/bridgehead-loop на бой на поле застройки. До готовности нового боя старый бой можно использовать только как временный reference/diagnostic.
5. После того как новый цикл работает, убрать или переименовать устаревшие сущности (`Garrison`, `Bridgehead`, старый castle battlefield).

- [ ] 1.1. `js/game/state.js` — единое дерево состояния:
  - `resources: { wood, stone, iron, crystal, gold }`
  - `waveNumber` (стартует с 1, доходит до 7)
  - `screen: 'top' | 'bottom'`
  - `buildField`: сетка 5×7, каждый тайл `{ x, y, occupant: null | 'obstacle' | { buildingId } }`
  - `buildings`: массив зданий `{ id, type, level, tiles: [{x,y}], hp, maxHp, cooldownTimer }`
  - `obstacleRemovalCost` (стартует с 3, +1 за каждое удаление)
  - `workers`: массив рабочих `{ id, level, location: 'pile' | { mineId, slotIndex }, position?: { x, y } }`
  - `workerBuyCost` (стартует с 5, +1 за каждую покупку)
  - `mines`: `{ wood, stone, iron, crystal }`, каждая `{ unlocked, slots: [{ workerId | null, progress }], productionIntervalSeconds: 5 }`
  - `mergeCapLevel` (стартует с 3, апгрейд поднимает до 7)
  - `workerStartLevel` (стартует с 1, апгрейд поднимает до 3)
  - `workerBuyDiscount` (мультипликатор, 1.0 по умолчанию)
  - `buildingBuyDiscount` (мультипликатор, 1.0 по умолчанию)
  - `unlockedBuildingTypes` (Set типов, доступных для покупки — начиная с базового набора)
  - `battle`: `{ active, enemies: [], allies: [], projectiles: [], mines: [], spawnTimer, enemiesToSpawn }`
  - `pendingUpgradeChoice: null | [upgradeA, upgradeB, upgradeC]`
- [ ] 1.2. `js/game/config.js` — загрузка JSON (`buildings.json`, `units.json`, `waves.json`, `mines.json`, `balance.json`).
- [ ] 1.3. `js/game/factories.js` — фабрики: `createBuilding`, `createWorker`, `createEnemy`, `createAlly`, `createProjectile`, `createMine`.
- [ ] 1.4. `js/game/systems/`:
  - `buildFieldSystem.js` — размещение/перемещение зданий, проверка занятости тайлов, удаление препятствий.
  - `buildingSystem.js` — покупка, апгрейд, генерация юнитов казармой/стрельбищем/конюшней/башней магов, стрельба турели, детонация мины.
  - `workerSystem.js` — покупка, мердж, назначение в слот, свободное перемещение внутри «кучи».
  - `mineSystem.js` — тик производства ресурсов, разблокировка шахт, добавление слотов.
  - `battleSystem.js` — цикл боя, спавн врагов, targeting, атаки, победа/поражение.
  - `upgradeSystem.js` — генерация 3 случайных апгрейдов после победы, применение выбранного.
- [ ] 1.5. `js/game/ui/`:
  - `topScreen.js` — панель ресурсов, поле застройки, кнопка «В бой», лента покупки зданий.
  - `bottomScreen.js` — панель ресурсов, зона шахт, зона кучи рабочих, кнопка покупки рабочего.
  - `screenSwitcher.js` — обработка свайпа вверх/вниз, анимация переключения.
  - `battleOverlay.js` — во время боя скрывает лишний UI, показывает поле боя.
  - `upgradeModal.js` — модалка выбора 1 из 3 апгрейдов.
- [ ] 1.6. `js/main.js` — запуск, game loop, `requestAnimationFrame`, роутинг событий.

Комментарий: коммитим модули маленькими шагами, каждый шаг — работоспособный (даже если UI-заглушка).

- [ ] 2.1. Разметка. Контейнер поля — CSS grid 5 колонок × 7 строк, тайл ~14–16 vw шириной для мобильного портрета.
- [ ] 2.2. Панель ресурсов сверху: 5 иконок + значения + «Волна N».
- [ ] 2.3. Генерация препятствий на старте: 10 случайных тайлов из 35 занять «деревом», остальные свободны. Штаб (3×2) размещается на свободных тайлах в нижней части поля (гарантированно свободных — при генерации обстаклов резервируем футпринт штаба).
- [ ] 2.4. Кнопка «удалить» у каждого препятствия. Цена = `obstacleRemovalCost` золота, кнопка disabled если золота недостаточно. После клика — снимаем препятствие, `obstacleRemovalCost += 1`.
- [ ] 2.5. Кнопка «улучшить» у каждого здания. Цена и ресурс берутся из `buildings.json` по типу и текущему уровню. Максимальный уровень 3 для всех кроме штаба.
- [ ] 2.6. Перетаскивание зданий пальцем (pointer events) между свободными тайлами. Подсветка валидных / невалидных позиций. Проверка футпринта здания.
- [ ] 2.7. Лента покупки зданий под кнопкой «В бой»:
  - горизонтальный скролл, зациклен (три копии массива подряд + виртуальная прокрутка, либо реальный looped scroll).
  - каждая карточка: иконка, цена, кнопка «купить» (enabled только если ресурсов хватает **и** на поле есть место под футпринт этого здания).
  - после покупки — здание размещается в случайное валидное место, лента обновляет доступность.
- [ ] 2.8. Кнопка «В бой» — запускает `battleSystem.start()`.

## 3. Нижний экран — производство ресурсов

- [ ] 3.1. Панель ресурсов сверху (та же, что и на верхнем — общий компонент).
- [ ] 3.2. Верхняя зона — 4 карточки шахт (дерево / камень / металл / кристаллы). Металл и кристаллы стартуют заблокированными (визуально «замок»).
- [ ] 3.3. У каждой шахты слоты рабочих (по 1 на старте, максимум 5, апгрейдится). Свободный слот — пустой круг; занятый — иконка человечка с уровнем.
- [ ] 3.4. Тик производства: у каждого занятого слота накапливается `progress`; когда `progress >= 5s`, начисляется X ресурса по уровню рабочего:
  - lvl1=2, lvl2=3, lvl3=5, lvl4=7, lvl5=10, lvl6=12, lvl7=15.
- [ ] 3.5. Нижняя зона — «куча» рабочих: свободное позиционирование (2D-контейнер с абсолютным позиционированием). Стартовые позиции при покупке — рандомные внутри зоны.
- [ ] 3.6. Кнопка «Купить рабочего»: цена = `floor(workerBuyCost × workerBuyDiscount)` золота, +1 к базовой цене за каждую покупку.
- [ ] 3.7. Механика мерджа: перетаскиваем рабочего на другого рабочего того же уровня → получаем рабочего на уровень выше. Ограничение — `mergeCapLevel`. Слоты шахт тоже участвуют в мердже.
- [ ] 3.8. Перетаскивание рабочих: между позициями в куче, между слотами разных шахт, из кучи в слот и обратно.

## 4. Бой

- [ ] 4.1. По кнопке «В бой» — скрываем панель ресурсов, номер волны, кнопку «В бой», ленту покупки. Поле застройки со зданиями остаётся видимым и превращается в поле боя. Свайп между экранами блокируется на время боя.
- [ ] 4.2. Спавн врагов: сверху экрана, интервал и количество по номеру волны:
  - W1: 3 врага, 3s; W2: 4/3s; W3: 5/2s; W4: 7/2s; W5: 10/1s; W6/W7 — экстраполировать (см. §7 в `waves.json`).
- [ ] 4.3. Каждый враг ищет ближайшее здание, движется к нему по прямой (простое steering, без pathfinding в MVP; препятствия обходятся с помощью проверки соседних тайлов и небольшого A* по свободным тайлам — можно упростить до «идти к ближайшему свободному соседнему тайлу цели»).
- [ ] 4.4. Здание-цель атакуется в контакте: враг наносит урон раз в 1 секунду.
- [ ] 4.5. Здания в бою:
  - **Пассивные** (стена, штаб) — просто препятствия/цели с HP.
  - **Турель** — раз в 2s стреляет по ближайшему врагу в радиусе (радиус в конфиге, ~3 тайла).
  - **Мина** — не атакуема; при наступании врагом наносит урон и уничтожается.
  - **Казарма / Стрельбище / Конюшни / Башня магов** — раз в N секунд создают союзного юнита рядом со зданием.
- [ ] 4.6. Союзные юниты: ищут ближайшего врага, идут к нему, атакуют в своём радиусе. Параметры — из `units.json` (см. §7).
- [ ] 4.7. Проекции (стрелы, магия) — простая летящая точка с фиксированной скоростью, наносит урон при попадании.
- [ ] 4.8. Победа: все враги волны уничтожены. Поражение: HP штаба ≤ 0.
- [ ] 4.9. После боя:
  - все разрушенные здания восстанавливаются на своих исходных тайлах с полным HP.
  - UI возвращается.
  - победа: `waveNumber += 1`, начисляется золото (по волне), открывается модалка апгрейдов.
  - поражение: начисляется утешительное золото (≈50% от победного), модалка апгрейдов не открывается, `waveNumber` не растёт.
- [ ] 4.10. После победы над волной 7 — экран «Prototype complete» + кнопка «Начать заново».

## 5. Апгрейды после победы

- [ ] 5.1. `upgradeSystem.rollChoices(state)` — выбирает 3 уникальных доступных апгрейда из списка:
  - `+1 slot mine wood` (если слотов < 5)
  - `+1 slot mine stone` (если слотов < 5)
  - `unlock mine iron` (если ещё не разблокирована)
  - `unlock mine crystal` (если ещё не разблокирована)
  - `+1 slot mine iron` (если разблокирована и слотов < 5)
  - `+1 slot mine crystal` (если разблокирована и слотов < 5)
  - `raise merge cap` (если `mergeCapLevel < 7`)
  - `raise worker start level` (если `workerStartLevel < 3`)
  - `worker discount -10%` (стакается)
  - `unlock new building` (если есть закрытые здания)
  - `building discount -10%` (стакается)
