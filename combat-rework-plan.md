# План реализации: переработка боевой системы

**Основание:** [GDD_combat_rework.md](GDD_combat_rework.md) (v1.0, 2026-07-08)
**Ветка:** `codex/combat-rework` (от `master`)
**Суть:** контрпики (треугольник юнитов), передний/задний ряд, летающие, камикадзе, сброс волны при поражении. Шахты, мердж, покупка юнитов — без изменений.

---

## 1. Сверка GDD с фактическим кодом

GDD писался по памяти об архитектуре — часть ссылок не совпадает с master. Реальная картина:

| В GDD | Реально в коде |
|---|---|
| `js/ui/screens/garrisonScreen.js` | Весь UI — один файл [ui.js](js/game/ui.js) (~1000 строк) + `index.html`. Модалки и переключатели делаем там |
| `state.garrison`, `state.bridgehead` | `state.bridgeheadUnits`. Отдельной «казармы» нет: юнит идёт резерв/шахта → плацдарм напрямую ([garrisonSystem.js:49](js/game/systems/garrisonSystem.js:49)) |
| `state.battle.phase = "lost"`, `currentWave` | `state.battle.status` (`idle/cooldown/fighting/retreating/siege/won`), `nextWaveIndex` / `activeWaveIndex` |
| «Проигрыш — неясное поведение» | Уже сейчас: юниты игрока гибнут безвозвратно, золото за убитых начисляется в момент смерти врага (`awardEnemyGold`). **Но** убитые враги волны запоминаются в `waveProgress.defeatedEnemyIndexesByWave` и при повторном заходе не респавнятся — это и есть «стачивание», которое GDD убирает |
| Не упомянуто | **Система экипировки** (weapon + armor из `equipment.json`): сейчас обязательна при выставлении на плацдарм и является единственным стоком ресурсов шахт (wood/ore/iron/crystal). → Подтверждено ГД: классы её полностью заменяют, стоимость переезжает в класс (Д1) |
| Не упомянуто | **Замок и осада**: победа в игре = снести замок после всех волн; при гибели отряда враги «отступают к замку» и выходят снова. → Подтверждено ГД: замок и осада уходят, остаются бои и волны (Д2) |
| `chooseTarget` | Есть `chooseClosestTarget` + `keepCurrentTarget` (липкость цели). Фильтр `canHitFlying` нужно применять **до обоих** |

Также два формата волн в самом GDD расходятся: в разд. 8–9 группы заданы ключами `{ "front": [...], "flying": [...] }`, в разд. 10.2 — массивом `groups: [{ formationRow, enemies }]`. **Принимаем формат 10.2**, `formationRow ∈ front | back | flying | kamikaze`.

## 2. Решения (подтверждены геймдизайнером 2026-07-09)

- **Д1 — классы полностью заменяют экипировку.** Вместо покупки weapon+armor покупается класс: ресурсная стоимость переезжает из `equipment.json` в `classes.json` (поле `costs` у каждого класса — сток ресурсов шахт сохраняется). Статы считаются только по GDD 10.3: `база уровня × мультипликаторы класса`, никаких дополнительных множителей гира. `equipment.json` и weapon/armor-код (селекторы в казарме, `getWeaponConfig`/`getArmorConfig`, поля юнита) выпиливаются из этой ветки. Цифры GDD (Рыцарь ур.3 = HP 110) валидны напрямую.
- **Д2 — замок и осада уходят.** Остаются только бои и волны: победа в игре = зачищены все волны. Сносим: `state.castle` и его HP/рендер, статусы `siege`/`won`-через-замок, атаку замка юнитами, retreat-к-замку механику, castle-бокс в физической арене, секцию `castle` в `balance.json`. «Сброс волны» при поражении больше не требует retreat-театра — враги просто уходят с поля, волна пересоздаётся заново.
- **Д3 — дискретные бои.** Бой закончился победой → выжившие телепортируются в гарнизон (на плацдарм) с полным HP; следующая волна — по кнопке. **Развилка (вернуться после реализации):** блокировать ли гарнизон/плацдарм на время боя. Если не блокировать — ростер может переполниться к возврату выживших, или наоборот появится контр-тактика спама юнитами в идущий бой, что убивает паззл-ощущение. Рабочее решение: **блокируем на период боя**, пометку не снимать до плейтеста.
- **Д4 — спавн-оффсеты рядов.** Оффсет ±15 из GDD в текущее поле не влезает (allySpawnX=8). Вопрос чисто визуальный — правим наживую при реализации: либо скалируем поле, либо подбираем позиции на глаз. Ориентир приёмки: задний ряд вступает в бой на ~2 с позже переднего.
- **Д5 — конфликт с experimental.md.** Идея №10 (класс на мердже L3, перки L3–L6) и «умный таргетинг» (front/mid/back со статбонусами) — другой дизайн той же зоны. В этой ветке следуем GDD: класс выбирается при покупке в казарме, ряда два, без статбонусов, таргетинг остаётся «ближайший». Идея №10 живёт отдельно.
- **Д6 — уровни и кап мерджа (подтверждено).** Таблица `unit-levels.json` перебалансируется под плоскую кривую GDD (+25% от базы за уровень: HP 32/40/48/56/64/72, атака 7/9/11/12/14/16). Кап мерджа: стартуем с **4**, после прохождения Локации 1 повышаем до **6**.
- **Д7 — камикадзе и летающие у игрока.** В таблице классов 10.1 их нет, в волнах Локаций 1–2 они только вражеские. Механики реализуем генерик (по полям `movementType`/`targetMode` на любом акторе), но игровых классов с ними пока не заводим.
- **Д8 — классы ярусов 3–4** (Отравитель, Паладин, Берсерк, Маг) — задел на Локацию 3. Делаем отдельным этапом после Фазы 4, чтобы не тормозить остальные шаги (см. Фазу 5).
- **Д9 — формат волн.** Расхождение форматов внутри GDD (разд. 8–9 vs 10.2) не принципиально — для прототипа берём наиболее удобный: `groups: [{ formationRow, enemies }]` из 10.2.

## 4. Чек-лист ручного плейтеста (после Фазы 3)

1. 3 Мечника → волна 1-1: победа без потерь или с минимальными.
2. 2 Мечника → волна 1-2: поражение → волна сбросилась на 100% HP, золото за убитых пришло, юниты потеряны.
3. Толпа Мечников → 1-5: поражение; 1 Рыцарь ур.3 → победа.
4. 1-4 без Лучника: орёл безнаказанно выносит отряд (мечники его не атакуют).
5. Камикадзе (2-3): убивается лучниками на подходе; при проходе — взрыв в заднем ряду.
6. Задний ряд визуально вступает в бой на ~2 с позже переднего.
7. Победа над волной → выжившие на плацдарме с полным HP, состав/ряды сохранены.
8. Во время боя казарма и плацдарм заблокированы (нельзя докупить класс или добавить юнита в идущий бой).
9. Замка на поле нет; зачистка последней волны = победа в игре.

Запуск: локальный статик-сервер из корня (`python -m http.server 8769` или tools), открыть `index.html`.
