# План реализации: переработка боевой системы

**Основание:** [GDD_combat_rework.md](GDD_combat_rework.md) (v1.0, 2026-07-08)
**Ветка:** `codex/combat-rework` (от `master`)
**Суть:** контрпики (треугольник юнитов), передний/задний ряд, летающие, камикадзе, сброс волны при поражении. Шахты, мердж, покупка юнитов — без изменений.

## 1. Сверка GDD с фактическим кодом

GDD писался по памяти об архитектуре — часть ссылок не совпадает с master. Реальная картина:

| В GDD | Реально в коде |
|---|---|
| `js/ui/screens/garrisonScreen.js` | Весь UI — один файл [ui.js](js/game/ui.js) (~1000 строк) + `index.html`. Модалки и переключатели делаем там |
| `state.garrison`, `state.bridgehead` | `state.bridgeheadUnits`. Отдельной «казармы» нет: юнит идёт резерв/шахта → плацдарм напрямую ([garrisonSystem.js:49](js/game/systems/garrisonSystem.js:49)) |
| `state.battle.phase = "lost"`, `currentWave` | `state.battle.status` (`idle/cooldown/fighting/retreating/siege/won`), `nextWaveIndex` / `activeWaveIndex` |
| «Проигрыш — неясное поведение» | Уже сейчас: юниты игрока гибнут безвозвратно, золото за убитых начисляется в момент смерти врага (`awardEnemyGold`). **Но** убитые враги волны запоминаются в `waveProgress.defeatedEnemyIndexesByWave` и при повторном заходе не респавнятся — это и есть «стачивание», которое GDD убирает |
| Не упомянуто | **Система экипировки** (weapon + armor из `equipment.json`): сейчас обязательна при выставлении на плацдарм и является единственным стоком ресурсов шахт (wood/ore/iron/crystal) |
| Не упомянуто | **Замок и осада**: победа в игре = снести замок после всех волн; при гибели отряда враги «отступают к замку» и выходят снова |
| `chooseTarget` | Есть `chooseClosestTarget` + `keepCurrentTarget` (липкость цели). Фильтр `canHitFlying` нужно применять **до обоих** |

Также два формата волн в самом GDD расходятся: в разд. 8–9 группы заданы ключами `{ "front": [...], "flying": [...] }`, в разд. 10.2 — массивом `groups: [{ formationRow, enemies }]`. **Принимаем формат 10.2**, `formationRow ∈ front | back | flying | kamikaze`.

## 2. Решения (зафиксировать до старта Фазы 1)

- **Д1 — экипировка остаётся (решение 2026-07-09).** GDD её не упоминает ни в «не меняется», ни в изменениях; по правилу «GDD — надстройка над существующей схемой, если не сказано иначе» экипировку сохраняем. Класс и гир складываются так: статы = `база уровня × мультипликаторы класса`, поверх — `weapon.attackMultiplier` / `weapon.attackSpeedMultiplier` / `armor.healthBonus`. **Боевой архетип определяет класс, не оружие**: `attackType`, `attackRange`, `splashRadius`, `canHitFlying` берутся из класса; поля `attackType`/`attackRangeBonus` оружия игнорируются (Лучник стреляет на 12 с любым оружием, Мечник с луком остаётся ближним). Сток ресурсов шахт остаётся на экипировке, классы бесплатны. Следствие: цифры GDD (Рыцарь ур.3 = HP 110) — baseline «голого» юнита; волны подгоняем на плейтесте с учётом среднего гира.
- **Д2 — замок.** Остаётся финальной целью (осада после последней волны). Retreat-анимация врагов остаётся как визуализация «сброса волны».
- **Д3 — дискретные бои.** После победы над волной выжившие возвращаются на плацдарм с полным HP (GDD 7.2 «выжившие юниты возвращаются в гарнизон»). Сейчас юниты остаются на поле и дерутся волну за волной с накопленным уроном — при таком режиме паззл-баланс волн («1 Рыцарь гарантированно берёт 1-5») не работает.
- **Д4 — спавн-оффсеты рядов.** Оффсет ±15 из GDD не влезает в поле: `allySpawnX = 8` (задний ряд ушёл бы в −7), `enemySpawnX = 78`, замок на 92 (задний ряд врагов — в замке). Ретюн в `balance.json`: союзники front/back = **23 / 8**, враги front/back = **70 / 85**. Подгонять под критерий приёмки №3 (задний ряд вступает в бой на ~2 с позже).
- **Д5 — конфликт с experimental.md.** Идея №10 (класс на мердже L3, перки L3–L6) и «умный таргетинг» (front/mid/back со статбонусами) — другой дизайн той же зоны. В этой ветке следуем GDD: класс выбирается при выставлении на плацдарм, ряда два, без статбонусов, таргетинг остаётся «ближайший». Идея №10 живёт отдельно.
- **Д6 — уровни и кап мерджа.** Таблица `unit-levels.json` перебалансируется под плоскую кривую GDD (+25% от базы за уровень: HP 32/40/48/56/64/72, атака 7/9/11/12/14/16). Кап мерджа: **4** в Локации 1, **6** после её прохождения (сейчас в `balance.json` стоит 6 без прогрессии).
- **Д7 — камикадзе и летающие у игрока.** В таблице классов 10.1 их нет, в волнах Локаций 1–2 они только вражеские. Механики реализуем генерик (по полям `movementType`/`targetMode` на любом акторе), но игровых классов с ними пока не заводим.

## 4. Чек-лист ручного плейтеста (после Фазы 3)

1. 3 Мечника → волна 1-1: победа без потерь или с минимальными.
2. 2 Мечника → волна 1-2: поражение → волна сбросилась на 100% HP, золото за убитых пришло, юниты потеряны.
3. Толпа Мечников → 1-5: поражение; 1 Рыцарь ур.3 → победа.
4. 1-4 без Лучника: орёл безнаказанно выносит отряд (мечники его не атакуют).
5. Камикадзе (2-3): убивается лучниками на подходе; при проходе — взрыв в заднем ряду.
6. Задний ряд визуально вступает в бой на ~2 с позже переднего.
7. Победа над волной → выжившие на плацдарме с полным HP, состав/ряды сохранены.

Запуск: локальный статик-сервер из корня (`python -m http.server 8769` или tools), открыть `index.html`.
