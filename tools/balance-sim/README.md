# Balance simulators

Инструменты для проверки баланса Fortress-режима. Чистый Python 3, без зависимостей
(`random`, `math`, `heapq`, `statistics`, читают `data/*.json`). Запуск из этой папки:
`python <файл>.py`.

> 📘 Ментальная модель, три метрики здоровья и таблица «симптом → диагноз → ручка» —
> в [`PLAYBOOK.md`](PLAYBOOK.md). Инвентарь конкретных числовых ручек — в [`PARAMS.md`](PARAMS.md).

## Схема (одна связка, 4 файла)

```
battle_sim.py   ── боевой движок (порт fortressBattleSystem + pathfinding), читает JSON live
      │
calibrate.py    ── общие загрузчики данных + specs_cost + winrate + min_defense
      │
   sim.py       ── ГЛАВНЫЙ оракул: связанный цикл mining↔battle (экономика + бой + награды)
                   overbuild.py  ── standalone: потолок капзатрат «переполненного» форта
                   roles.py      ── standalone: RPS-поверхность оборона×тип-волны (текстура боя)
```

| Скрипт | Вопрос | Что печатает |
|---|---|---|
| **`sim.py`** | Как живёт вся петля на текущем балансе? | по волнам: prep/battle-время, win%, buy$, ростер, demand, shift%, gold; итог — LOOP/PACING/DIFFICULTY/DYNAMICS/ECONOMY/REWARDS. Это единственный источник истины для петли. |
| **`overbuild.py`** | Может ли кривая цены зданий (level-sum) перекрыть перепроизводство? | капзатраты целевого «умеренного» и «максед» форта на сетке 29 тайлов vs добыча за игру. Моделирует то, что `sim.py` НЕ видит (игрок строит МИНИМУМ обороны, а не оверилд). |
| **`roles.py`** | Награждает бой правильный МИКС или это плоский DPS-порог? | матрица win% сборок×типов-волн + вердикт (specialize / contested band). Для проверки «бинарности» боя. |
| `battle_sim.py` | (библиотека) | Импортируется `sim.py`, `calibrate.py`, `roles.py`. Напрямую не запускается для отчёта. |
| `calibrate.py` | (библиотека) | Загрузчики + стоимости + win% для `sim.py` и `roles.py`. |

## Быстрый старт

```bash
cd tools/balance-sim
python sim.py            # вся петля на ТЕКУЩИХ данных (baseline)
python sim.py sweep      # грид по калибровочным ручкам (CFG) без правки JSON
python overbuild.py      # потолок оверилда vs добыча
python roles.py          # RPS-поверхность боя
```

## Как крутить параметры

- **Все числа — в `data/*.json`** (единственный источник истины). Скрипты читают их live.
- **Свипы без правки JSON:** правь словарь `CFG` в `sim.py` (или гоняй `python sim.py sweep`) —
  `prod_scale`, `battle_mul`, `shift_base`, `exp`, `cost_scale`, `gold_scale`, `repair_scale`,
  `rest_mult`. Плюс модульные глобалы `sim.YIELD_PP`, `sim.START_GOLD`, `sim.ECAP_*`.
- **Награды за волны** моделируются в `sim.py` (`pick_reward`, флаг `REWARDS_ON`) — точный порт
  live-драфта (1 Perm + 1 Temp + 1 OneShot за победу; компаунд perm-карт).
- **Боёвку** меняешь в `data/fortress-{waves,enemies,buildings,units}.json` — `battle_sim.py`
  и `roles.py` подхватят автоматически.

## Важные оговорки (границы модели)

- **`sim.py` строит ЛИН-оборону** (минимум, чтобы пройти волну) — он НЕ моделирует оверилд
  игрока, который топит в ресурсах и застраивает всю сетку. Реальный фейсролл от оверилда виден
  только в плейтесте; `overbuild.py` считает его потолок аналитически.
- **`sim.py` не моделирует стоимость абилок** (ability-accumulation sink) и **`calibrate.py`
  specs_cost стоимости зданий устарели** (count-based, а игра перешла на level-sum) — defense-spend
  в симе занижен, но это малый член.
- **`battle_sim.py` — ПОРТ, а не общий код.** Он читает `data/*.json`, но боевую ЛОГИКУ
  (armor/splash/pathing) надо руками держать в синхроне с `fortressBattleSystem.js`.
- **Экономические политики бота — приближение «нормального игрока»**, не оптимум. Абсолютные
  prep-числа читай как порядок величины; сравнения (ручка vs ручка, волна vs волна) — надёжные.
- **Один прогон `sim.py` = один сид** (ролл ростера, сиды боя, драфт наград). Для стабильной
  картины гоняй несколько сидов или читай медиану — драфт наград особенно шумный на одном сиде.
