# Balance simulators

Инструменты для проверки баланса Fortress-режима. Чистый Python 3, без зависимостей
(`random`, `math`, `heapq`). Запуск из этой папки: `python <файл>.py`.

> 📘 **Новичку в балансе проекта — начни с [`PLAYBOOK.md`](PLAYBOOK.md):** ментальная
> модель, три метрики здоровья, таблица «симптом → диагноз → ручка», как читать вывод.
> Этот README — про запуск, `PARAMS.md` — про конкретные числа.

`battle_sim.py` — **порт боевого движка 1:1** с `js/game/systems/fortressBattleSystem.js`
+ `pathfinding.js` (сетка 5×7, A*, спавны, юниты, турели, трап-мины, проджектайлы,
коллизии, условия победы/поражения). Числа продублированы из `data/*.json` вручную.

> ⚠️ **Синхронизация:** это ПОРТ, а не общий код. При правках `data/*.json` или боевой
> логики — обновляй константы в шапках `battle_sim.py` (`FUNITS`, `BUILD`, `WAVES`) и
> в экономических скриптах, иначе симуляция разойдётся с игрой. Числа-кандидаты см. `PARAMS.md`.

## Что каким скриптом отвечать

| Скрипт | Вопрос | Что печатает |
|---|---|---|
| `difficulty.py` | Какая оборона проходит волну N и сколько стоит? | win% + средний остаток HQ + время боя + стоимость (д/о/ж/к) по лестнице сборок, для каждой волны. Выявляет «нужную силу» и **бинарность боя**. |
| `econ_timing.py` | Сколько времени добычи требует каждая волна на ТЕКУЩЕМ балансе? | prep-время (стойла), ростер, добыча в/о/ж/к по волнам. Показывает затыки текущей игры (волны 3/6/7 = 300–600с). |
| `pacing_v2.py` | Убирает ли новый каркас стойла? | S-свип (подбор масштаба цены под добычу) + детальный прогон на рекомендованном `COST_SCALE`. 20 волн, экспонента 1.05, награда только из боя. |
| `battle_sim.py` | (библиотека) | Импортируется `difficulty.py`. Напрямую не запускается для отчёта. |

## Быстрый старт (завтра, на свежую голову)

```bash
cd tools/balance-sim
python difficulty.py     # кривая «нужной силы» + бинарность боя на ТЕКУЩИХ данных
python econ_timing.py    # где стойла добычи на ТЕКУЩИХ данных
python pacing_v2.py      # предлагаемый каркас: S-свип + прогон на COST_SCALE=12
```

## Как крутить параметры

- **Текущий баланс vs предлагаемый** и все ручки — в `PARAMS.md`.
- `pacing_v2.py`: правь `reward(w)`, `COST_SCALE`, функцию `cost(w)` (шкалы ресурсов и
  волны ввода железа/кристалла), `EXP`/`BASE` (экспонента покупки), `N_WAVES`.
- `difficulty.py`: правь `LADDER` (сборки зданий и позиции) и `BUY_COST`/`UP_COST`.
- Боёвку меняешь в `battle_sim.py` — `FUNITS` (юниты/враги), `BUILD` (здания/уровни),
  `WAVES` (составы волн).

## Важные оговорки

- **Числа-кандидаты в `pacing_v2.py` — ДО калибровки.** Финальные значения цены/награды/
  добычи тюним ПОСЛЕ ввода системы шахтёров (трейты + скудные слоты + крутая добыча по
  уровням), т.к. под изменённую кривую добычи калибруется цена. См.
  `plans/balance-and-systems-redesign.md` §3, §8.
- **Бой в игре имеет случайные позиции спавна и завалы** → скрипты усредняют по сидам
  (`difficulty.py` — 15–25 сидов на точку).
- **Экономические политики бота — приближение «нормального игрока»**, не оптимум.
  Абсолютные prep-числа читай как порядок величины, а сравнения (текущий vs предлагаемый,
  волна vs волна) — как надёжные.
- `pacing_v2.py` включает фикс «гарантировать рабочего на каждую нужную шахту» — без него
  жадная политика морит новую низкоуровневую шахту (артефакт бота, не игры).
